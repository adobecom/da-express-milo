# Remote Branch Workflow

This reference covers all procedures specific to **remote-branch-mode**
(`.aem.live` preview URL). It is loaded on demand by the main skill
when `remote-branch-mode = true`.

> **SAFETY — single rule, stated once:**
> NEVER push commits to `main` or `stage`. Every push targets the
> feature branch only. Verify the branch name before every push.

> **Repository scope:**
> All branches are created on **`adobecom/da-express-milo`**.
> The branch-prefixed EDS URL only resolves branches on
> `adobecom/da-express-milo`. Never push to forks.

---

## 1. Publishing a `.aem.page` URL

Parse the URL to extract org, repo, and page path:
```
https://main--<repo>--<org>.aem.page/<path>
```

### Path safety check (BLOCKING)

- **Path contains `/drafts/`:** safe to publish automatically.
- **Path does NOT contain `/drafts/`:** present this warning and
  wait for explicit confirmation:
  ```
  WARNING: Publishing this page will make it live on production.
  Path: <path>
  Are you sure you want to proceed? (y/n)
  ```
  Do NOT call the publish API without passing this check.

### Publish via EDS admin API

```bash
# Preview first
curl -s -w "\n%{http_code}" -X POST \
  "https://admin.hlx.page/preview/<org>/<repo>/main/<path>"

# Then publish
curl -s -w "\n%{http_code}" -X POST \
  "https://admin.hlx.page/live/<org>/<repo>/main/<path>"
```

Expect **200 OK** for both. On success, the published URL is:
```
https://main--<repo>--<org>.aem.live/<path>
```

On failure (401/403), ask the user to check permissions or publish manually.

## 2. Branch creation

All git operations target `adobecom/da-express-milo` via the
**`origin`** remote (confirmed via `git remote -v`).

1. **Determine branch name**:
   - If the user explicitly requested a specific feature branch,
     use that name.
   - Otherwise, default to `<ldap>-<block-name>-autogenerated`
     (e.g. LDAP `methomas`, block `ax-hero` → `methomas-ax-hero-autogenerated`).

2. **Check for existing branch**:
   ```bash
   git ls-remote origin refs/heads/<branch-name>
   ```
   If the branch already exists, inform the user and ask how to proceed:
   - Delete and recreate it
   - Use a different name
   - Reuse the existing branch

3. **Determine the base branch**:
   - Default: `stage`.
   - If the user specified a custom base branch, use that instead.
     Check whether it exists on `adobecom/da-express-milo`:
     ```bash
     git ls-remote origin refs/heads/<base-branch>
     ```

4. **Create the feature branch** from the base branch:
   ```bash
   git fetch origin <base-branch>
   git push origin origin/<base-branch>:refs/heads/<branch-name>
   ```

5. **Inform the user** which branch will receive all code changes
   and **wait for confirmation**:
   ```
   All code will be pushed to branch: <branch-name>
   Repository: adobecom/da-express-milo
   Base branch: <base-branch>
   Branches "main" and "stage" will NOT be modified.
   Proceed? (y/n)
   ```

> **STOP**: Do NOT proceed until the user confirms the branch.

## 3. Pushing code to the feature branch

Use **git CLI** to commit and push to the feature branch on
`adobecom/da-express-milo`.

### Workflow

1. **Fetch and check out the feature branch** from origin:
   ```bash
   git fetch origin <branch-name>
   git checkout -b <branch-name>-work origin/<branch-name>
   ```
   Use a temporary local branch name (e.g. `<branch-name>-work`) to
   avoid conflicts with any existing local branch.

2. **Create or copy the block files** into the working tree:
   - `express/code/blocks/<name>/<name>.js`
   - `express/code/blocks/<name>/<name>.css`

3. **Commit and push**:
   ```bash
   git add express/code/blocks/<name>/
   git commit -m "Add <block-name> block (autogenerated)"
   git push origin HEAD:<branch-name>
   ```

4. **Clean up** the temporary local branch:
   ```bash
   git checkout <previous-branch>
   git branch -D <branch-name>-work
   ```

For subsequent pushes (visual fixes, a11y fixes), repeat the same
pattern: fetch, checkout, apply changes, commit, push, clean up.

After the push succeeds, the code is available at the branch-prefixed URL:
```
https://<branch>--da-express-milo--adobecom.aem.live/express/code/blocks/<name>/<name>.js
```

## 4. CDN force-refresh after push

The CDN may cache stale JS/CSS from the feature branch.
After every push, force-refresh:

### 4a. Trigger EDS code preview for changed files

For each file you pushed, call the EDS code preview API:

```bash
curl -s -X POST \
  "https://admin.hlx.page/code/adobecom/da-express-milo/<branch>/express/code/blocks/<name>/<name>.js"
curl -s -X POST \
  "https://admin.hlx.page/code/adobecom/da-express-milo/<branch>/express/code/blocks/<name>/<name>.css"
```

> **NEVER use a wildcard or broad path** for code preview.
> Wildcard cache clears invalidate files across the entire organization.
> Always reference the specific files that changed.

### 4b. Bypass browser HTTP cache in Playwright

```js
await page.route('**/*.{js,css}', (route) => {
  route.continue({
    headers: {
      ...route.request().headers(),
      'cache-control': 'no-cache',
    },
  });
});
```

Set this route **once** at the start of the Playwright session.
It persists across navigations and reloads.

### 4c. Reload and verify

1. Navigate to the Playwright URL (`https://<branch>--da-express-milo--adobecom.aem.live/<page-path>`).
2. Hard-reload the page.
3. Wait for the block to fully render before proceeding.

If files still 404 after force-refresh, wait 10-15 seconds and
repeat from 4a.

## 5. Batched iteration strategy

Each verification requires a push + CDN refresh. To avoid excessive
commits and API load, **batch multiple fixes into a single push**:

1. After each screenshot comparison, collect all issues found.
2. Apply fixes to the local files (JS and/or CSS).
3. Before pushing, review the accumulated changes and assess
   whether the code is in a high-confidence state: correct syntax,
   consistent token usage, all breakpoints addressed, no obvious
   regressions from prior fixes.
4. Only push when you believe the batch of fixes is coherent and
   likely to produce a meaningful visual improvement. A single
   push covering 3-5 fixes is preferable to 5 separate pushes.
5. After pushing, repeat the force-refresh procedure (section 4),
   then take the new screenshot.

Use a descriptive commit message summarizing the batch, e.g.
"Fix heading alignment, tablet spacing, and CTA padding".
